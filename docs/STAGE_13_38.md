# Этап 13.38. ReAct-автозаполнение слотов

## Цель

Добавить детерминированный слой заполнения слотов из read-only ReAct-вызовов. Один вызов может вернуть несколько полей, а администратор решает, какие из них становятся слотами сценария.

## Что сделано

- Добавлен конфигурационный домен `slot_autofill_profiles`.
- В схемах слотов появился способ заполнения `slot_autofill`.
- В Admin UI добавлен раздел `Сценарии обработки -> 0.1 Автозаполнение слотов`.
- Профиль исполняется только в статусе `active`; отдельного пользовательского переключателя включения нет.
- В профиле автозаполнения можно:
  - выбрать схему слотов;
  - выбрать read-only ReAct-вызов;
  - настроить источники входных параметров вызова;
  - привязать поле результата к существующему слоту;
  - создать новый слот из поля результата.
- Обязательность поля результата в профиле задается независимо от `required` в общем контракте ReAct-вызова.
- В Operator UI первый шаг показывает результаты ReAct-автозаполнения и значения слотов, заполненные этим способом.

## Правило безопасности

Runtime не создает новые слоты. Новый слот создается только в draft-конфигурации администратором, проходит валидацию и активируется как часть схемы слотов.

## Отличие от разрешения атрибутов

`slot_autofill_profiles` предназначены для детерминированного заполнения по контракту ReAct-вызова.

`attribute_resolution_profiles` остаются для неоднозначных случаев, где нужны несколько шагов обогащения, LLM-правило, уточнение у клиента или эскалация оператору.
