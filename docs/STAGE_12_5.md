# Этап 12.5: сценарная модель оркестратора

Этап 12.5 делает сценарий обращения основной единицей управления в консоли администратора и видимой рабочей картой в панели оператора. Группа `Сценарии обработки` связывает слоты, классификацию, ReAct-политику, матрицу запуска инструментов, prompt pack и правила эскалации.

## Реализовано

- Новые домены Config Registry:
  - `service_scenarios`;
  - `slot_schemas`;
  - `classification_routes`;
  - `orchestrator_policy`;
  - `tool_launch_matrix`;
  - `prompt_packs`;
  - `escalation_policies`.
- JSON Schema для каждого нового домена в `contracts/config`.
- Default-конфигурация для сценариев:
  - `password_reset`;
  - `software_issue`;
  - `hardware_issue`;
  - `network_issue`;
  - `access_request`;
  - `unknown`.
- Cross-validation связей:
  - сценарий ссылается на существующие slot schema, route, policy, prompt pack и escalation policy;
  - required slot должен иметь вопрос или источник автозаполнения;
  - порядок вопросов проверяется по приоритету "кто -> что -> когда";
  - route ссылается на существующий workflow state;
  - запуск ReAct-вызова ссылается на существующий tool, подключение и операцию;
  - parameter bindings ссылаются на существующие slots;
  - action tool нельзя включить в `execution_level=auto`, если policy инструмента требует согласование;
  - prompt pack обязан содержать 7 блоков;
  - escalation package обязан содержать полный набор полей для передачи в канал эскалации.
- Admin API:
  - `GET /admin/scenarios`;
  - `GET /admin/scenarios/{scenario_id}`;
  - `POST /admin/scenarios/{scenario_id}/simulate`.
- Operator API:
  - `GET /operator/scenarios`;
  - `GET /operator/scenarios/{scenario_id}`;
  - `POST /operator/scenarios/{scenario_id}/simulate`.
- Раздел консоли администратора `Сценарии обработки -> Сценарии`:
  - выбор сценария через выпадающее меню без отдельной таблицы "Карта сценариев";
  - операции `Создать`, `Модифицировать`, `Удалить`;
  - формы редактирования для создания и модификации сценария;
  - выбор подготовленных блоков через выпадающие поля: схема слотов, маршрут, ReAct-политика, матрица инструментов, пакет промптов и политика эскалации.
- Подразделы консоли администратора с сохраненными номерами шагов:
  - `Сценарии обработки -> 0. Слоты`;
  - `Сценарии обработки -> 1. Разрешение атрибутов`;
  - `Сценарии обработки -> 2. Классификация и маршрут`;
  - `Сценарии обработки -> 3. ReAct-планирование`;
  - `Сценарии обработки -> 4. Инструменты и матрица запуска`;
  - `Сценарии обработки -> 5. Решение и эскалация`;
  - `Сценарии обработки -> 6. Промпты`.
- В шаге `0. Слоты`:
  - добавление, изменение и удаление отдельных слотов без ручного JSON;
  - существующие слоты свернуты по умолчанию, удаление доступно справа в компактной строке, новый слот раскрывается сразу для заполнения;
  - технические идентификаторы сценария, схемы слотов, маршрута, политик, запусков инструментов и prompt pack не показываются как поля управления, но сохраняются в payload для связей, логов и audit trail;
  - пояснения к основным полям слота: `slot_id`, `display_name`, `priority_group`, `required`, `fill_method`, а также контекстным полям `user_question`, `case_source_ref`, `extraction_instruction`, `resolution_profile_id`, `fallback_question`, `operator_hint`.
  - override-пороги уверенности слота скрыты в свернутом блоке, включаются отдельным чекбоксом и используются только для исключений; когда override выключен, UI показывает системные значения как неактивную базу.
- В шаге `3. ReAct-планирование`:
  - системные пороги уверенности редактируются явно;
  - сценарные override-пороги находятся в свернутом блоке и включаются чекбоксом `Включить`;
  - пороги применяются к slot filling, LLM extraction, уточнениям и передаче оператору.
- Подразделы `0-5` работают как каталоги подготовленных блоков:
  - блоки не содержат управляемой связи со сценариями;
  - у каждого блока есть `Название`;
  - доступны операции `Создать`, `Модифицировать`, `Удалить`;
  - блок `Где используется` показывает сценарии, выбравшие этот блок;
  - удалить блок нельзя, пока он используется сценарием или схемой слотов.
- Раздел консоли администратора `Сценарии обработки -> 6. Промпты`:
  - выбор пакета промптов через выпадающее меню;
  - операции `Создать`, `Модифицировать`, `Удалить`;
  - редактирование блока `Пакет промптов: обязательные блоки`;
  - семь обязательных блоков prompt pack;
  - защита от удаления пакета, если на него ссылается активный сценарий.
- Панель оператора:
  - одно поле ввода неструктурированного текста заявки;
  - выбор сценария;
  - галочка `Тестовый прогон сценария` для включения dry-run проверки слотов и маршрута;
  - выбор режима тестового прогона: `Проверка конфигурации`, `С моделью`, `С моделью и безопасными интеграциями`, `Отладочный запуск с подтверждениями`;
  - раскрываемые разрешения прогона: вызов модели, read-only интеграции, mock-интеграции, action-вызовы только с подтверждением;
  - раскрытая карта пяти шагов оркестратора;
  - вопросы обогащения только по слотам, где вопрос действительно настроен или возвращен профилем разрешения;
  - автоматические недостающие слоты без вопроса показываются как ожидающие извлечения/автозаполнения и не требуют ручного ответа оператора;
  - в режиме `С моделью` dry-run вызывает настроенный OpenAI-compatible/LiteLLM endpoint для `llm_extraction`, показывает значение, confidence, решение порога и причину;
  - трасса тестового прогона показывает выполненные, пропущенные и заблокированные шаги;
  - отображение готовых и заблокированных запусков инструментов по матрице.

## Архитектурные решения

- `service_scenarios` является верхним уровнем управления поведением AI.
- `service_scenarios` собирает сценарий из независимых блоков `0-5`; сами блоки не выбирают сценарии.
- Существующие `tools`, `integration_endpoints`, `workflow_states`, `workflow_transitions`, `prompts`, `model_routing` и `n8n_workflows` остаются нижнеуровневыми справочниками.
- Операции создания, модификации и удаления сценариев идут через `service_scenarios` и общий процесс `draft/validate/regression/activate`.
- Редактирование каждого шага идет через свой домен Config Registry и общий процесс `draft/validate/regression/activate`; сценарий только выбирает нужную запись домена.
- Отдельный preview системного промпта в UI убран: обязательные блоки prompt pack являются редактируемым шаблоном в `Сценарии обработки -> 6. Промпты`.
- В UI матрицы инструментов используется единый "Режим исполнения сейчас"; backend сохраняет его в `execution_level` и `target_execution_level`.
- Эффективные пороги уверенности наследуются в порядке `система -> сценарий -> слот`; профиль разрешения атрибута может добавить внутренние пороги только для исключений своего resolver.
- Пакет промптов хранится структурно и не редактируется как один свободный текст.
- Тестовый прогон сценария вынесен из консоли администратора в операторский UI и работает как dry-run без запуска интеграций.
- Тестовый прогон имеет безопасные уровни глубины: базовый режим не вызывает LLM и интеграции, а расширенные режимы включаются оператором явно.
- Операторская панель читает сценарную модель через отдельные operator endpoint'ы, а не через административные permissions.
- Перестроение базы знаний остается явным действием оператора и отражено в RBAC роли `operator`.

## Ограничения этапа

- Сценарная модель используется административной консолью, операторской панелью и dry-run обогащением; полный runtime-переход core workflow на эти домены остается следующим шагом.
- Поле `source` в слоте заменено на способ заполнения и выбор готового профиля разрешения атрибута.
- Базовый тестовый прогон сценария не вызывает LLM и внешние системы; режим `С моделью` вызывает только LLM для заполнения слотов, а action-вызовы остаются за подтверждением.
- В MVP action tools остаются за подтверждением оператора, даже если вид запуска отмечен как `auto`.
