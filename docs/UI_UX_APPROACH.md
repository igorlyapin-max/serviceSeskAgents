# UX-подход для административной консоли

Этот документ фиксирует общий подход к интерфейсам консоли администратора. Его нужно применять не только к `1. Разрешение атрибутов`, но и ко всем разделам, где администратор управляет повторяемыми сущностями: сценариями, слотами, ReAct-вызовами, интеграциями, привязками операций, маршрутами и правилами эскалации.

## Цель

Администратор должен видеть связную рабочую модель, а не набор больших технических форм. Интерфейс должен помогать быстро понять:

- какие объекты уже настроены;
- в каком порядке они применяются;
- что будет выполнено;
- где объект используется;
- что нужно изменить прямо сейчас.

## Базовый паттерн

Для повторяемых сущностей используется схема:

```text
Компактный список объектов
  -> один раскрытый редактор выбранного объекта
  -> расширенные настройки под свернутым блоком
```

По умолчанию показывается не большая форма, а компактный список или таблица. Редактируется только один выбранный объект. Остальные объекты остаются строками списка.

## Правила проектирования

### 1. Сначала обзор, потом редактирование

Если в разделе есть несколько однотипных объектов, верхний уровень должен быть сводной таблицей или списком:

- номер или порядок;
- название;
- тип / роль;
- ключевая связь;
- статус;
- действия `Изменить` и `Удалить`.

Большие карточки для всех объектов сразу не используются.

### 2. Один активный редактор

Кнопка `Изменить` раскрывает редактор только выбранной строки. Это снижает визуальный шум и не заставляет администратора скроллить через чужие настройки.

Новый объект после добавления раскрывается автоматически.

### 3. Основные поля отделены от расширенных

В обычном режиме показываются только поля, которые нужны для большинства изменений.

Редко используемые настройки уходят в блок:

```text
Дополнительно >
```

К расширенным настройкам относятся:

- технические описания;
- ручные override-поля;
- поля совместимости;
- низкоуровневые параметры;
- диагностические значения;
- будущие настройки, которые не нужны в MVP каждый день.

### 4. Таблица вместо растянутых строк

Если пользователь сопоставляет параметры, условия или поля результата, используется плотная таблица:

```text
Параметр | Тип / обязательность | Значение взять из
```

Каждая строка должна быть самостоятельной и визуально короткой. Подстрочники используются только для важных пояснений, а не для каждого поля.

### 5. Человекочитаемые значения важнее технических ссылок

В интерфейсе основным текстом показываются понятные названия:

- `Слот: Логин пользователя`;
- `Выходной слот: Email руководителя`;
- `Сущность users: email`;
- `ReAct-вызов: Поиск пользователя в AD`;
- `Endpoint-операция: Найти пользователя`.

Технические ссылки вида `slot:user_login`, `entity:users.email`, `search_ad_users` могут быть показаны вторым слоем: мелким текстом, tooltip или в диагностике.

### 6. Автозаполнение вместо обязательного ручного ввода

Если значение можно вывести из выбранного объекта, UI должен предлагать его автоматически:

- имя сущности результата из ReAct-вызова;
- поля результата из `result_schema`;
- обязательные параметры из контракта;
- порядок нового шага;
- безопасный статус черновика.

Пользователь может переопределить значение, но не должен начинать с пустой формы без подсказок.

### 7. Контракты и JSON не являются основным интерфейсом

JSON может оставаться контрольной точкой или диагностикой, но основной режим редактирования должен быть формой с ограниченными вариантами.

Если объект основан на JSON Schema, схема остается источником истины, а UI строит из нее управляемую форму.

### 8. Связи показываются рядом с объектом

Для объектов, которые могут использоваться в других местах, нужен блок `Где используется`.

Удаление используемого объекта должно быть заблокировано понятным сообщением:

```text
Нельзя удалить: используется в сценарии ...
```

### 9. Не показывать внутренние идентификаторы как бизнес-поля

ID остаются в payload, логах и диагностике. В обычном интерфейсе пользователь выбирает название, сценарий, слот, ReAct-вызов или endpoint-операцию из списка.

Исключение: создание нового объекта, где технический ключ действительно нужен до появления сущности. Даже в этом случае поле должно быть вторичным.

### 10. Компактность важнее декоративности

Консоль администратора является рабочим инструментом. Для нее предпочтительны:

- плотные таблицы;
- короткие строки;
- предсказуемые кнопки;
- раскрывающиеся детали;
- минимальные отступы внутри повторяемых элементов;
- отсутствие вложенных больших карточек.

## Применение к блоку `Обогащение контекста`

Целевой вид:

```text
Шаги обогащения

N | Шаг | ReAct-вызов | Результат | При ошибке | Действия
1 | Поиск пользователя | search_ad_users | users | продолжить к LLM | Изменить / Удалить
2 | Поиск руководителя | get_manager | manager | эскалировать оператору | Изменить / Удалить

Редактор выбранного шага
  Основное
  Параметры вызова
  Контракт результата >
  Дополнительно >
```

Поля результата не раскрываются по умолчанию. UI показывает краткую строку:

```text
Контракт результата: 4 поля
```

При смене ReAct-вызова параметры и поля результата пересобираются из контракта выбранного вызова.

## Критерии качества

Интерфейс считается удачным, если администратор может:

- понять весь список объектов без скролла через длинные формы;
- изменить один объект, не теряя контекст остальных;
- выбирать связи из списков, а не вводить технические идентификаторы вручную;
- видеть, что будет сохранено, без чтения JSON;
- раскрывать сложные детали только когда они действительно нужны.
