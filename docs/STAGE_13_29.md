# Этап 13.29: визуальный редактор контрактов endpoint-операций

## Цель

Снизить количество ошибок при настройке mock endpoint и других endpoint-операций.

Администратор не должен вручную синхронизировать:

- `request_schema.properties`;
- `request_schema.required`;
- `response_schema.properties`;
- `response_schema.required`;
- `mock_output`.

## Решение

Основной режим редактирования контракта операции перенесен в UI:

- входные параметры операции редактируются строками формы;
- поля ответа операции редактируются строками формы;
- тестовый ответ `mock` строится из полей ответа операции;
- JSON остается контрольной точкой и экспертным режимом.

Backend-контракты не меняются. На сохранение по-прежнему уходит обычный JSON:

- `request_schema`;
- `response_schema`;
- `mock_output`.

## UI

В карточке операции endpoint добавлены блоки:

- `Входные параметры операции`;
- `Поля ответа операции`;
- `Тестовый ответ mock`;
- `JSON контрольная точка`.

Список операций подключения скрыт в раскрываемом блоке `Операции подключения`.
Это оставляет настройки подключения компактными, когда администратор меняет только URL, авторизацию или статус endpoint.

Для входных параметров и полей ответа поддерживаются:

- имя поля;
- название;
- тип;
- обязательность;
- описание;
- дополнительные ограничения.

Поддерживаемые типы MVP:

- `string`;
- `integer`;
- `number`;
- `boolean`;
- `array`;
- `object`.

Для массивов поддерживается тип элемента:

- `string`;
- `integer`;
- `number`;
- `boolean`;
- `object`.

## Mock-ответ

Для endpoint с adapter type `mock` UI строит форму тестового ответа из `response_schema`.

Правила:

- обязательные поля ответа должны быть заполнены;
- тип значения проверяется на стороне UI до сохранения;
- итоговый `mock_output` генерируется из формы;
- итоговый JSON дополнительно валидируется backend через существующий механизм.

## JSON контрольная точка

JSON-блок остается доступен в карточке операции:

- `JSON входа`;
- `JSON ответа`;
- `JSON тестового ответа`;
- кнопка `Применить JSON в форму`.

Обычный поток работы:

1. Администратор меняет форму.
2. UI обновляет JSON контрольную точку.
3. Сохранение активирует итоговый JSON через текущий draft/validate/regression/activate flow.

Экспертный поток:

1. Администратор меняет JSON.
2. Нажимает `Применить JSON в форму`.
3. UI разбирает поддерживаемую часть схемы и обновляет форму.

## Ограничения MVP

- Глубокий визуальный редактор вложенных объектов не реализован.
- Для массива объектов используется `items.type=object` и `additionalProperties=true`.
- Сложные конструкции JSON Schema (`anyOf`, `oneOf`, `allOf`, `$ref`) отображаются предупреждением и требуют экспертного режима.

## Связь с ReAct-вызовом

> Обновление: на этапе 13.30 переключатель синхронизации заменен прямым редактором видимого контракта ReAct. Актуальная модель описана в [STAGE_13_30.md](STAGE_13_30.md).

В актуальном UI `Привязка операций` показывает все параметры endpoint-операции и поля endpoint-ответа.

Администратор выбирает:

- какие endpoint-параметры видит ReAct как `parameters_schema`;
- какие endpoint-параметры заполняются константой или секретом;
- какие поля endpoint-ответа возвращаются в `result_schema`;
- какие имена этих параметров и полей будут видны оркестратору.

## Критерии выхода

- В UI есть визуальное редактирование входных параметров операции.
- В UI есть визуальное редактирование полей ответа операции.
- Для `mock` есть форма тестового ответа.
- JSON остается видимым как контрольная точка.
- Сохранение endpoint продолжает использовать существующую backend-валидацию JSON Schema.
- Smoke-проверка фиксирует наличие новых UI-блоков.
