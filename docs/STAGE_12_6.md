# Этап 12.6: профили разрешения атрибутов

> Обновление: целевая модель профилей переработана на этапе 13.24. Этот документ фиксирует исторический дизайн этапа 12.6; актуальная структура описана в [STAGE_13_24.md](STAGE_13_24.md).

Этап 12.6 выносит логику заполнения слотов в отдельный каталог профилей разрешения атрибутов. Схема слотов выбирает готовое правило заполнения, а профиль описывает resolver: признаки, операцию разрешения, правила оценки результата операции, матрицу решений, уточняющие вопросы и fallback.

## Реализовано

- Новый контракт `contracts/config/attribute-resolution-profiles.schema.json`.
- Новый домен Config Registry `attribute_resolution_profiles`.
- Default-профили:
  - `Поиск логина в AD по ФИО`;
  - `Устройство пользователя из AD`;
  - `Устройство из CMDB`;
  - `Подсеть по локации из CMDB`;
  - `Пользователь запроса доступа из AD`;
  - `История успешных сбросов пароля`.
- Read-only tool `search_ad_users` и подключения `mock` / `n8n`.
- Раздел Admin UI `Сценарии обработки -> 1. Разрешение атрибутов`.
- CRUD-операции профилей через `draft/validate/regression/activate`.
- Список профилей может быть пустым: `profiles: []` является валидной конфигурацией, если сценарии не используют профили разрешения атрибутов.
- Профиль редактируется формой без JSON-поля и без сводной таблицы профилей.
- Внутренние пороги профиля скрыты в свернутом блоке `Пороги внутри профиля`; пустые значения наследуются от эффективных порогов слота, сценария и системы.
- Редактор слотов использует `Способ заполнения` и `Профиль разрешения атрибута` вместо прямого выбора технического источника.
- Dry-run сценария возвращает `attribute_resolution` с операцией разрешения, сводкой результата и решением матрицы.
- Операторская панель показывает способ заполнения и решение resolver-профиля в шаге `1. Приём и нормализация`.

Дополнение этапа 12.7 упрощает эту модель: профиль разрешения атрибута трактуется как детерминированный resolver, а не как mini-workflow с ручными переходами.

## Цель

- Убрать hardcoded список источников слота из UI и backend.
- Заменить поле `source` в слоте на выбор способа заполнения и профиля разрешения атрибута.
- Дать администратору отдельное меню для создания, модификации и удаления профилей.
- Оставить tools runtime-сущностью исполнения, а не основной сущностью настройки слота.

## Раздел UI

- Пункт меню: `Сценарии обработки -> 1. Разрешение атрибутов`.
- Операции:
  - `Создать`;
  - `Модифицировать`;
  - `Удалить`.
- UI-обвязка:
  - выпадающий выбор профиля;
  - форма создания;
  - форма модификации;
  - блоки признаков, операции разрешения, оценки результата, матрицы решений, уточнения и передачи человеку;
  - подтверждение удаления;
  - сохранение через `draft/validate/regression/activate`;
  - audit event для каждой операции.

## Профиль разрешения атрибута

Профиль содержит:

- название;
- статус;
- сценарий для выбора слотов в UI;
- целевой слот, выбираемый из доступных слотов сценария;
- выходные слоты, выбираемые из доступных слотов сценария;
- признаки для поиска;
- операция разрешения;
- правила оценки результата операции;
- матрицу решений;
- правила уточняющих вопросов;
- уточняемые атрибуты и пакет передачи человеку, выбираемые в UI из доступных слотов сценария;
- лимит попыток;
- внутренние confidence thresholds профиля;
- fallback;
- audit/log флаги.

## Изменение сценариев

В редакторе слота остается:

- `Способ заполнения`:
  - вопрос пользователю: редактируется `user_question`;
  - из данных обращения: редактируется `case_source_ref`;
  - извлечение моделью: редактируется `extraction_instruction` и необязательные `examples`;
  - профиль разрешения атрибута: выбирается `resolution_profile_id`, при необходимости задается `fallback_question`;
  - ручное заполнение оператором: редактируется `operator_hint`.
- Нерелевантные поля способа заполнения скрываются в UI и не должны попадать в payload.
- Override-пороги уверенности слота находятся в свернутом блоке и нужны только для исключений.

Для выбранного профиля UI показывает read-only сводку: выходы, fallback, audit и возможные уточняющие вопросы.

## Пример

Профиль `Поиск логина в AD по ФИО`:

1. Извлечь ФИО из текста обращения через LLM.
2. Найти пользователя в AD.
3. Если найден один пользователь и confidence выше порога, заполнить `user_login`.
4. Если найдено несколько пользователей, запросить должность, подразделение или табельный номер.
5. Повторить поиск с уточнением.
6. Если результат неоднозначен после лимита попыток, передать оператору.

## Валидации

- Слот может ссылаться только на существующий профиль, который выдает этот слот как output.
- В UI администратора `target_slot_id` и `output_slots` выбираются из схемы слотов выбранного сценария. Если в существующей конфигурации осталось значение вне выбранного сценария, UI показывает его как отсутствующий слот, чтобы администратор явно исправил настройку.
- `clarification_policy.ask_for_attributes` и `handoff_policy.package` в UI также выбираются из схемы слотов выбранного сценария. Выбранные слоты автоматически объявляются как атрибуты профиля, если они еще не были объявлены.
- Шаги обогащения контекста ссылаются на существующие ReAct-вызовы с привязкой к endpoint-операциям.
- Входные параметры ReAct-вызовов ссылаются только на входные слоты, выходные слоты, предыдущие сущности, константы или секреты.
- `clarification_policy` содержит вопрос и уточняемые признаки.
- History step содержит фильтры статусов, качества, периода и разрешенных полей.
- Fallback задан для неоднозначного и неуспешного пути.
- Слот не может ссылаться на профиль, не выдающий именно этот слот как output.

## Критерии выхода

- В админской консоли есть раздел `1. Разрешение атрибутов` с CRUD.
- В разделе нет сводки профилей и JSON-редактора шагов; порядок поведения профиля настраивается через карточки.
- В сценариях слот выбирает готовый профиль, а не технический источник данных.
- Pipeline поиска `login_name` для сброса пароля описывается конфигурацией.
- История заявок используется только через явно настроенный профиль.
- Тестовый прогон показывает шаги разрешения атрибутов и вопросы уточнения без реального вызова внешних систем.
